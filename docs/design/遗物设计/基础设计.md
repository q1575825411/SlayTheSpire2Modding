# 遗物设计

当前仓库尚未沉淀具体遗物清单；这一阶段先明确技术落点，避免后续重复改入口。

## 当前技术方案

- 使用 `RitsuLibFramework.CreateContentPack(Entry.ModId)` 做显式注册。
- 当前角色体系下的遗物进入 `SoldierRelicPool`，作为 `士兵` 角色内容。
- 具体遗物统一继承 `Scripts/Relics/MyFirstRelic.cs`。
- 单个遗物补齐后，需要同时登记：
  - `Scripts/Relics/<遗物名>.cs`
  - `MyFirstStS2Mod/localization/zhs/relics.json`
  - `MyFirstStS2Mod/images/relics/<遗物名>.png`
  - `Entry.Init()` 里的 `.Relic<SoldierRelicPool, <遗物名>>()`

## 事件涉及

根据教程与当前 RitsuLib 口径，遗物设计里要区分两套信息：

- `真实实现类别`：`Starter / Common / Uncommon / Rare / Shop / Event / Ancient`
- `设计标签`：`联动遗物 / 事件专属 / 官府线 / 豪杰线 / 黄巾线` 等

注意：

- `联动遗物` 不是正式 `RelicRarity`，只能作为设计标签使用。
- 事件直接产出、且不进入普通遗物池的遗物，优先写成 `Event`。
- 仍会进入普通遗物池的遗物，才继续使用 `Common / Uncommon / Rare` 等常规类别。

后续每个新遗物设计稿，建议补齐这些字段：

- `真实实现类别：<Rarity>`
- `设计标签：无 / 联动遗物 / 事件专属 / <事件线名>`
- `常规遗物池：是 / 否`
- `事件涉及：无 / 可涉及 / 已联动`

若写为 `已联动`，还要继续补齐：

- `关联事件：<事件名>`
- `涉及方式：前置条件 / 分支条件 / 奖励产出 / 事件移除 / 互斥条件`
- `失去遗物后的影响：联动结束 / 分支关闭 / 无额外处理`

当前已经可以作为事件涉及候选的遗物方向：

- `身份牌`：适合进入身份分支事件或阵营判断事件。
- `官印`：适合进入商店、官府、交易类事件。
- `虎符`、`阵图`、`战鼓`：适合进入军阵、行军、兵备类事件。

## 单个遗物最小模板

```csharp
public sealed class ExampleRelic : MyFirstRelic
{
    public override RelicRarity Rarity => RelicRarity.Common;
}
```
